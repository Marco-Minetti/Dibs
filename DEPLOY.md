# Deploying the dibs backend (HTTPS)

You need three external services plus a host. All have free tiers good enough for testing.

| Need | Service | Free tier |
|------|---------|-----------|
| Host + Postgres + HTTPS | **Render** (blueprint included) | yes |
| Photo storage (S3) | **Cloudflare R2** (or AWS S3) | 10 GB free |
| Email login codes | **Resend** | 3k emails/mo |
| Payments | **Stripe** | test mode free |

## 1. Deploy the API on Render

1. Push this repo to GitHub.
2. Render ‚Üí **New + ‚Üí Blueprint** ‚Üí select the repo. It reads `render.yaml`, creates `dibs-db` (Postgres) and `dibs-api` (web service), runs migrations, and assigns an HTTPS url like `https://dibs-api.onrender.com`.
3. That url is your **`DIBS_API_BASE`** for the app.

`JWT_SECRET` and `CODE_PEPPER` are generated by Render. `DATABASE_URL` is wired automatically. Health check: `/health`.

> Until you add a Resend key, login codes don't email ‚Äî they print in the Render logs (fine for your own testing).

## 2. Photo storage ‚Äî Cloudflare R2

1. Cloudflare ‚Üí R2 ‚Üí create bucket `dibs-uploads`.
2. Create an R2 API token (Object Read & Write). Note the **Account ID**, **Access Key ID**, **Secret**.
3. Enable a public dev domain (or a custom domain) for the bucket ‚Üí that's your `CDN_BASE_URL`.
4. In Render set: `S3_ENDPOINT=https://<accountid>.r2.cloudflarestorage.com`, `S3_REGION=auto`, `S3_BUCKET=dibs-uploads`, `S3_ACCESS_KEY_ID`, `S3_SECRET_ACCESS_KEY`, `CDN_BASE_URL=https://<your-r2-public-domain>`.

(For AWS S3 instead: leave `S3_ENDPOINT` blank, set the real region, and make the bucket/CloudFront serve `CDN_BASE_URL`.)

## 3. Email ‚Äî Resend

1. Resend ‚Üí API Keys ‚Üí create one ‚Üí set `RESEND_API_KEY` in Render.
2. Verify a sender domain, then set `EMAIL_FROM="dibs <login@yourdomain>"`.

## 4. Payments ‚Äî Stripe (test mode)

1. Stripe dashboard (test mode) ‚Üí Developers ‚Üí API keys. Set in Render: `STRIPE_SECRET_KEY=sk_test_‚Ä¶`, `STRIPE_PUBLISHABLE_KEY=pk_test_‚Ä¶`.
2. Webhook: Developers ‚Üí Webhooks ‚Üí add endpoint `https://<your-api>/api/payments/webhook`, events `payment_intent.succeeded` and `payment_intent.payment_failed`. Copy the signing secret ‚Üí `STRIPE_WEBHOOK_SECRET=whsec_‚Ä¶`.
3. Local testing alternative: `stripe listen --forward-to localhost:4000/api/payments/webhook` prints a `whsec_‚Ä¶`.
4. Test card: `4242 4242 4242 4242`, any future expiry, any CVC.

**Seller payouts:** the buyer's money lands in your platform Stripe account until a seller connects a payout account. Hitting `POST /api/payments/connect` returns a Stripe Connect onboarding link; once a seller finishes it, their sales route to them automatically (minus `PLATFORM_FEE_BPS`).

## 5. Point the app at the backend

In `mobile/` build the web bundle with your live url (see `BUILD_ANDROID.md`):

```bash
DIBS_API_BASE=https://dibs-api.onrender.com \
DIBS_STRIPE_PK=pk_test_xxx \
npm run build:web
```

## Smoke test after deploy

```bash
curl https://<your-api>/health
curl "https://<your-api>/api/schools?q=new"
curl -X POST https://<your-api>/api/auth/request-code \
  -H 'content-type: application/json' -d '{"email":"you@your.edu"}'
# code arrives by email (or appears in the Render logs in dev)
```

## Other hosts
The backend is a standard Node app with a `Dockerfile`, so **Railway**, **Fly.io**, or **Heroku** work too ‚Äî provision Postgres, set the same env vars, run `npm run migrate` once, then `npm start`. Only the blueprint file is Render-specific.
